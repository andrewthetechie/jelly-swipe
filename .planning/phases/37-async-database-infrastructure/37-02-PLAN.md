<objective>
Move database startup responsibility out of the FastAPI app process and into an explicit bootstrap wrapper that runs migrations first, initializes the async runtime, and then starts Uvicorn.

Purpose: Phase 37 is not complete until the running app stops depending on in-process schema/bootstrap work and instead assumes the database is already migrated and ready before request handling begins.

Output:
- `jellyswipe/bootstrap.py` entrypoint for migration-first startup
- `__init__.py` lifespan reduced to app concerns and async runtime disposal
- `db.py` compatibility shims that delegate Phase 36 maintenance onto the async path
- `README.md` and Docker startup guidance aligned to the bootstrap wrapper
- infrastructure and DB smoke tests updated for the new startup contract
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator environment -> bootstrap wrapper | deployment configuration decides which database receives migrations and runtime initialization |
| bootstrap wrapper -> ASGI server | the server must not start until the database is migrated and runtime initialization succeeds |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-04 | Tampering | `jellyswipe/bootstrap.py` DB target selection | mitigate | resolve the target through one `DATABASE_URL`-first helper path and reuse it for both Alembic and runtime initialization |
| T-37-05 | Denial of Service | startup path | mitigate | keep migration execution in the bootstrap wrapper and fail fast before Uvicorn serves requests when `upgrade_to_head(...)` errors |
| T-37-06 | Denial of Service | async runtime lifecycle | mitigate | initialize the engine explicitly before server start and `await dispose_runtime()` on shutdown so stale engines do not leak across process lifetime |
</threat_model>
