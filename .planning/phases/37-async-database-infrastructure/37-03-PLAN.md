Refactor `tests/conftest.py` so the fixture layer mirrors the new runtime order:

- add one shared helper, `_bootstrap_temp_db_runtime(db_path, monkeypatch)`, that each fixture can reuse. For each temp DB file it must:
  - computes the sync Alembic URL with `build_sqlite_url(db_path)`
  - computes the runtime async URL with `build_async_sqlite_url(db_path)`
  - runs `upgrade_to_head(sync_url)`
  - initializes the async runtime for that temp DB before app creation
- in that helper/fixture teardown, always call `await dispose_runtime()` (or the sync wrapper needed by the fixture shape) before clearing overrides/singletons so the process-global cached engine/sessionmaker cannot remain bound to the prior temp DB
- when building `test_config` for `create_app(...)`, include:
  - `"DATABASE_URL": sync_database_url` where `sync_database_url = build_sqlite_url(db_path)` so tests use the same canonical contract as normal startup
  - `"DB_PATH": <filesystem path>` only for the remaining sync compatibility helpers
  - `"TESTING": True`
  - `"SECRET_KEY": os.environ["FLASK_SECRET"]`
- do not store the async `sqlite+aiosqlite:///...` URL in `test_config["DATABASE_URL"]`; derive it separately inside the helper and pass it only to the runtime initializer

Update `db_connection`, `app`, `app_real_auth`, and `client_real_auth` to use or consume that shared bootstrap path instead of open-coding `upgrade_to_head(...)` and standalone `DB_PATH` patching in multiple places. Preserve:

- `dependency_overrides.clear()` in teardown
- provider singleton reset in teardown
- rate limiter reset behavior
- `set_session_cookie()` exactly as-is

Make the teardown order explicit in the fixture code/comments: dispose runtime first, then clear overrides and other test singletons, so a later fixture can reinitialize against a second temp DB without stale engine state.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pytest fixture setup -> temp database state | test bootstrap mutates per-test databases and runtime globals |
| test app fixtures -> dependency/provider overrides | auth/provider overrides can leak state across tests if teardown is incomplete |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-37-07 | Tampering | `tests/conftest.py` fixture bootstrap | mitigate | provision a fresh temp DB per test through `upgrade_to_head(...)` and set the runtime URL per fixture instance |
| T-37-08 | Repudiation | dependency override teardown | mitigate | keep explicit `dependency_overrides.clear()` and provider/rate-limiter reset in fixture teardown so test outcomes stay attributable |
| T-37-09 | Denial of Service | runtime reuse across tests | mitigate | call `dispose_runtime()` in shared fixture teardown and verify reinitialization across at least two temp DBs so one global engine cannot stay pointed at the wrong file |
</threat_model>
