<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI → GitHub repository | Workflow has read access to code, write access via GITHUB_TOKEN |
| CI → PyPI/uv | Installs dependencies from PyPI or uses cached uv.lock |
| CI → pytest | Executes test code with full filesystem access in runner |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | S (Spoofing) | GitHub Actions workflow | accept | Workflow uses standard actions (checkout@v4, setup-python@v5) from trusted GitHub org; no third-party actions |
| T-17-02 | T (Tampering) | uv sync --frozen | accept | Uses frozen lockfile (uv.lock) committed to repo; tampering would require commit access |
| T-17-03 | I (Information Disclosure) | pytest coverage output | accept | Coverage reports show line numbers of production code; this is intended for developer visibility, not a security issue |
| T-17-04 | D (Denial of Service) | Test execution | mitigate | Added 10-minute timeout to job prevents hanging CI from consuming runner resources indefinitely |
| T-17-05 | E (Elevation of Privilege) | GITHUB_TOKEN | accept | Workflow has default permissions (contents: read for checkout); no write operations performed |
| T-17-06 | R (Repudiation) | Workflow execution logs | accept | GitHub Actions maintains immutable execution logs; all test results are auditable |
</threat_model>
