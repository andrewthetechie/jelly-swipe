# Phase 31 — UI Design Contract

> Visual and interaction contract for Phase 31: FastAPI App Factory and Session Middleware.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Scope Assessment

**This phase has no new UI surfaces.** Phase 31 is a pure backend refactor: it replaces the
Flask factory header in `jellyswipe/__init__.py` with a FastAPI app factory, wires
`SessionMiddleware`, registers middleware (`ProxyHeadersMiddleware`, `RequestIdMiddleware`),
and configures `XSSSafeJSONResponse` as the default response class. No new HTML, CSS, or
JavaScript is introduced. The existing `jellyswipe/templates/index.html` and
`jellyswipe/static/styles.css` are unchanged.

The one template interaction is the migration from `render_template("index.html")` to
`Jinja2Templates(directory="jellyswipe/templates").TemplateResponse(...)` — the rendered
HTML output is byte-for-byte identical from the browser's perspective.

The design contract below documents the **existing** visual system as the governing contract
so the executor does not inadvertently change rendered output while porting the factory.

**Phase 31 executor note:** The Jinja2 `TemplateResponse` call must use `request=request`
as the first positional context key (Starlette requirement) in addition to `media_provider`.
Omitting `request` will cause a Starlette runtime error and break the login screen render.

## Response Header Contract

These headers are part of the application's security surface and must appear on all HTTP
responses produced by the FastAPI app after Phase 31:

| Header | Value pattern | Implementation |
|--------|---------------|----------------|
| `X-Request-Id` | `req_{unix_ts}_{4-byte hex}` | `RequestIdMiddleware` injects via `request.state.request_id` |
| `Content-Security-Policy` | Existing policy from `add_csp_header()` verbatim | Added in `RequestIdMiddleware.dispatch()` or a separate `@app.middleware("http")` |

**XSS escaping contract:** `XSSSafeJSONResponse.render()` must escape `<` → `<`,
`>` → `>`, and `&` → `&` in all JSON output. This is the v1.5 behavior that
must be preserved (FAPI-04 success criterion 3).

## Middleware Stack Order

Starlette applies middleware in LIFO (last-added, first-to-process-request) order.
The executor must add middleware in this sequence so the effective processing order is correct:

| Add order | Middleware | Effective request order |
|-----------|-----------|------------------------|
| 1st added | `RequestIdMiddleware` | 3rd (innermost — sees request after session is decoded) |
| 2nd added | `SessionMiddleware` | 2nd |
| 3rd added | `ProxyHeadersMiddleware` | 1st (outermost — rewrites X-Forwarded headers first) |
