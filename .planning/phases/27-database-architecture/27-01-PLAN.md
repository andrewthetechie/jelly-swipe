<objective>
Enable WAL mode in SQLite to eliminate file-lock contention, and refactor the SSE generator to hold one persistent connection per client session instead of opening and closing a connection every 1.5-second poll cycle.

Purpose: Under concurrent SSE load, SQLite's default journal mode (rollback) causes file-lock contention because every poll cycle opens a new connection and the WAL file locks. WAL mode removes this bottleneck, and holding a persistent connection eliminates the per-cycle open/close overhead.

Output: Modified db.py (WAL PRAGMA statements), modified __init__.py (persistent SSE connection), updated test_db.py (WAL assertion test).
</objective>

       ```python
       def generate():
           last_genre = None
           last_ready = None
           last_match_ts = None
           POLL = 1.5
           TIMEOUT = 3600

           # Per DB-02: Hold one persistent connection for the entire
           # stream lifetime instead of opening/closing per poll cycle.
           # WAL mode (set in init_db) eliminates file-lock contention
           # so concurrent readers don't block this connection.
           conn = sqlite3.connect(jellyswipe.db.DB_PATH)
           conn.row_factory = sqlite3.Row
           try:
               deadline = time.time() + TIMEOUT
               while time.time() < deadline:
                   try:
                       row = conn.execute(
                           'SELECT ready, current_genre, solo_mode, last_match_data FROM rooms WHERE pairing_code = ?',
                           (code,)
                       ).fetchone()

                       if row is None:
                           yield f"data: {json.dumps({'closed': True})}\n\n"
                           return

                       ready = bool(row['ready'])
                       genre = row['current_genre']
                       solo = bool(row['solo_mode'])
                       last_match = json.loads(row['last_match_data']) if row['last_match_data'] else None
                       match_ts = last_match['ts'] if last_match else None

                       payload = {}
                       if ready != last_ready:
                           payload['ready'] = ready
                           payload['solo'] = solo
                           last_ready = ready
                       if genre != last_genre:
                           payload['genre'] = genre
                           last_genre = genre
                       if match_ts and match_ts != last_match_ts:
                           payload['last_match'] = last_match
                           last_match_ts = match_ts

                       if payload:
                           yield f"data: {json.dumps(payload)}\n\n"

                       time.sleep(POLL)
                   except GeneratorExit:
                       return
                   except Exception:
                       time.sleep(POLL)
           finally:
               conn.close()
       ```

       Key changes:

       - The `with get_db_closing() as conn:` block is removed — the query runs directly on the persistent `conn`
       - `finally: conn.close()` ensures the connection is closed on generator exit, including GeneratorExit and Exception
       - All state comparison logic (payload building) remains unchanged — just dedented out of the `with` block

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-01 | S (Spoofing) | SSE persistent connection | accept | Connection is read-only (SELECT); no authentication change from current design |
| T-27-02 | T (Tampering) | WAL mode write behavior | mitigate | WAL mode uses write-ahead log — concurrent writes are serialized by SQLite's built-in locking; synchronous=NORMAL maintains crash recovery guarantees per SQLite documentation |
| T-27-03 | I (Info Disclosure) | Connection lifecycle | accept | No new data exposed; persistent connection reads same data as per-cycle connections |
| T-27-04 | D (Denial of Service) | Persistent connection leak | mitigate | `finally: conn.close()` guarantees connection cleanup on generator exit, including GeneratorExit and unhandled exceptions; timeout (3600s) bounds generator lifetime |
| T-27-05 | E (Elevation of Privilege) | init_db PRAGMA execution | accept | init_db is called once at app startup; PRAGMAs are safe, documented SQLite operations |
</threat_model>
