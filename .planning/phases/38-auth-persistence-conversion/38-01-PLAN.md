<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser session cookie -> `request.session` | untrusted client-owned session data enters server auth handling here |
| auth service -> auth repository | trusted auth semantics cross into persistent auth-session storage |
| dependency layer -> protected routes | route authorization trusts `AuthUser` values returned by `require_auth` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-01 | Spoofing | `require_auth` / `auth_sessions` lookup | mitigate | resolve identity only through persisted `session_id` lookup, never from headers or client JSON, and raise the exact 401 on missing rows |
| T-38-02 | Tampering | local session state on stale session/logout | mitigate | clear the full session dict with `clear_session_state(...)` before 401 or logout completion so `active_room` and `solo_mode` cannot linger after auth failure |
| T-38-03 | Repudiation | best-effort logout cleanup failure | mitigate | swallow delete failures only after local clear and emit the `auth_session_delete_failed` log record so failures remain observable |
| T-38-04 | Denial of Service | request-driven expired-session cleanup | accept | keep cleanup bounded to a single `DELETE ... WHERE created_at < cutoff` during session creation; no unbounded scans or background scheduler are added in this phase |
</threat_model>
