    2. **Create `generate_request_id()` function** — generates a RequestId in format `req_<unix_timestamp>_<8-char-hex>` using `int(time.time())` for the timestamp and `secrets.token_hex(4)` for the random component. This follows the format specified in ERR-01.

    4. **Create `make_error_response(message, status_code, include_request_id=True, extra_fields=None)` helper** — builds a consistent error response:
       - Returns `jsonify({'error': message, **({'request_id': get_request_id()} if include_request_id else {}), **(extra_fields or {})})` with the given `status_code`
       - For 5xx: always uses `"Internal server error"` as the message (no exception details)
       - For 4xx: uses the provided specific client-safe message
       - Includes `request_id` field in all responses
       - Supports `extra_fields` dict for route-specific fields like `'cast': []` in get_cast

    5. **Create `log_exception(exc, context=None)` helper** — logs full exception details:
       - Uses `app.logger.error()` for 5xx exceptions, `app.logger.warning()` for 4xx
       - Logs: `request_id` (via get_request_id()), `route` (request.path), `method` (request.method), `exception_type` (type(exc).__name__), `exception_message` (str(exc)), `stack_trace` (traceback.format_exc())
       - Uses `extra={}` dict pattern consistent with http_client.py logging

    6. **Add `@app.before_request` hook** — calls `generate_request_id()` and stores result in `request.environ['jellyswipe.request_id']`. This ensures every request has a RequestId available.

    7. **Add RequestId to responses via existing `@app.after_request` hook** — modify the existing `add_csp_header` function to also set `response.headers['X-Request-Id'] = get_request_id()`. This adds the RequestId to every HTTP response for client-side debugging.

    IMPORTANT: Place all helper functions BEFORE route definitions but AFTER the app/variable setup section. Place the before_request hook right after the helpers.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Flask routes | Untrusted input from HTTP clients; exception details must not leak back |
| Flask routes → app.logger | Trusted boundary; full exception details acceptable in server-side logs |
| app.logger → log storage | Operator-accessible; RequestId enables correlation without exposing internals |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | I (Info Disclosure) | All route exception handlers | mitigate | Replace str(e) with generic messages; only log_exception() sees full details server-side |
| T-25-02 | I (Info Disclosure) | make_error_response helper | mitigate | 5xx responses always use "Internal server error" regardless of actual exception |
| T-25-03 | R (Repudiation) | RequestId generation | mitigate | Unique per-request RequestId enables audit trail correlation between client reports and server logs |
| T-25-04 | S (Spoofing) | X-Request-Id header | accept | RequestId in response header is for debugging only; not used for auth or authorization decisions |
</threat_model>
