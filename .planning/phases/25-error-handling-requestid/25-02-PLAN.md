<feature>
  <name>Error Handling and RequestId Tests</name>
  <files>tests/test_error_handling.py</files>
  <behavior>
    **RequestId Generation Tests (ERR-01):**
    - generate_request_id() returns string starting with "req_"
    - generate_request_id() returns string matching pattern req_<digits>_<hex>
    - Two calls to generate_request_id() return different values
    - RequestId is accessible from request context during route handling

    **RequestId Propagation Tests:**
    - HTTP response includes X-Request-Id header
    - X-Request-Id value matches req_<digits>_<hex> format
    - Multiple requests get different X-Request-Id values
    - X-Request-Id is included in error response body as request_id field

    **Error Response Format Tests (ERR-04):**
    - 404 response from get_trailer has format {"error": "...", "request_id": "req_..."}
    - 500 response from get_trailer has format {"error": "Internal server error", "request_id": "req_..."}
    - 404 response from get_cast includes "cast" field and request_id
    - 500 response from get_cast includes "cast" field and request_id
    - 500 response from add_to_watchlist has correct format
    - 500 response from get_server_info has correct format
    - 401 responses include request_id
    - 400 responses include request_id

    **Error Sanitization Tests (ERR-03):**
    - 500 response from get_trailer does NOT contain the original exception message
    - 500 response from get_cast does NOT contain the original exception message
    - 500 response from add_to_watchlist does NOT contain the original exception message
    - 500 response from get_server_info does NOT contain the original exception message
    - AST scan of __init__.py confirms zero str(e) in return statements

    **Error Logging Tests (ERR-02):**
    - Exception in get_trailer triggers app.logger.error with request_id
    - Exception in get_trailer triggers log with exception_type and exception_message
  </behavior>
  <implementation>
    Create tests/test_error_handling.py with the following structure:

    1. **Flask app fixture** — following the pattern from test_tmdb_auth.py (module reload + env var setup)

    2. **class TestRequestIdGeneration** — Unit tests for generate_request_id():
       - Test format matches `req_<digits>_<8-char-hex>` via regex
       - Test uniqueness across calls
       - Test timestamp component is recent (within last few seconds)

    3. **class TestRequestIdPropagation** — Integration tests using Flask test client:
       - Test X-Request-Id header present in responses
       - Test request_id field present in error response bodies
       - Test different requests get different IDs

    4. **class TestErrorSanitization** — Verify no str(e) leakage:
       - Mock get_provider to raise exceptions, verify response body does NOT contain exception message
       - AST scan of __init__.py source to verify no `str(e)` in return statements

    5. **class TestErrorResponseFormat** — Verify consistent format:
       - Test 4xx responses include specific message + request_id
       - Test 5xx responses include "Internal server error" + request_id
       - Test routes with extra_fields (get_cast includes 'cast' in error response)

    6. **class TestErrorLogging** — Verify structured logging:
       - Use caplog to verify app.logger.error called with request_id on 5xx
       - Verify log record includes route, method, exception_type
